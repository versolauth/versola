# fix(auth): serialize concurrent password changes (#92)

Closes #92

## Problem
`PostgresPasswordRepository.create` reads password history, checks reuse, then
inserts — all in one READ COMMITTED transaction. Two concurrent changes for the
same user can both read the old history, both pass the reuse check, and both
insert, bypassing the reuse/history-size constraints.

## Why `FOR UPDATE` (as suggested in the issue) isn't enough
Under READ COMMITTED it doesn't close the race:
- **Phantom:** `FOR UPDATE` locks only returned rows; after unblocking, the
  second txn re-reads the same rows and still misses the first txn's new insert.
- **Empty history:** the `SELECT` returns 0 rows, so nothing is locked — also
  breaks prune.

## Fix
Take a per-user transaction advisory lock before reading history:
```scala
sql"SELECT 1 FROM pg_advisory_xact_lock($PasswordHistoryLockNamespace, hashtext(${userId.toString}))"
  .query[Int].run()
```
The second txn waits for the first to commit, then reads a snapshot that
includes its insert. Fixes reuse, empty-history, and prune. Lock auto-releases
on commit/rollback. Namespace constant `= 92` scopes the lock per feature.

## Tests
Two parallel-change tests in `PasswordRepositorySpec` (real Postgres): same
password → exactly one succeeds + one `PasswordReuseError`; empty history →
`historySize` respected. Both fail on `main` and with `FOR UPDATE`-only.

## Notes
- `hashtext` is internal but stable; can swap if preferred.
- No migration — code-only.
